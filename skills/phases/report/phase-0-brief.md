# Phase 0 — Brief & Audience
# Aşama 0 — Brifing ve Hedef Kitle

## Gate Rule / Geçiş Kuralı

**EN:** Commissioner must approve the scope defined here before evidence gathering begins. Scope creep after Phase 1 starts is a primary cause of report delays and incoherence.

**TR:** Kanıt toplamaya başlamadan önce komisyoncu burada tanımlanan kapsamı onaylamalıdır. Aşama 1 başladıktan sonra kapsam genişlemesi, rapor gecikmelerinin ve tutarsızlığının birincil nedenidir.

---

## 1. Audience Analysis / Hedef Kitle Analizi

**EN:** Identify the primary reader(s). A report written for a policymaker differs fundamentally from one for an academic peer or a corporate board. Specify:

**TR:** Birincil okuyucu(ları) belirleyin. Bir politika yapıcı için yazılmış rapor, akademik bir meslektaş veya kurumsal yönetim kurulu için yazılmış rapordan temelden farklıdır. Belirtin:

```
Primary reader / Birincil okuyucu: ___
Reader's role / Okuyucunun rolü: ___
Technical literacy level / Teknik okuryazarlık düzeyi: [expert / practitioner / general]
What does the reader already know? / Okuyucu ne biliyor zaten?: ___
What must the reader NOT have to look up? / Ne araştırmak zorunda kalmamalı?: ___
```

**Reader types / Okuyucu türleri:**
- Policymaker / Politika yapıcı — needs recommendations first, evidence second
- NGO / STK — needs actionable findings, resource constraints acknowledged
- Academic / Akademik — expects methodology transparency, caveats
- Corporate / Kurumsal — needs ROI framing, risk quantification
- Media / Medya — needs headline findings, no jargon

---

## 2. Decision Context / Karar Bağlamı

**EN:** A report exists to inform a decision. If no decision is at stake, reconsider the format. Name the decision explicitly:

**TR:** Bir rapor bir kararı desteklemek için vardır. Eğer ortada bir karar yoksa biçimi yeniden düşünün. Kararı açıkça adlandırın:

```
Decision to be informed / Desteklenecek karar:
Decision maker / Karar verici:
Decision timeline / Karar zaman çizelgesi:
How will findings be used? / Bulgular nasıl kullanılacak?:
```

---

## 3. Scope Boundaries / Kapsam Sınırları

**EN:** Define scope both positively (what IS in) and negatively (what is NOT). The negative definition prevents scope creep and manages reader expectations.

**TR:** Kapsamı hem olumlu (neyin İÇİNDE olduğu) hem de olumsuz (neyin dışında olduğu) biçimde tanımlayın. Olumsuz tanım, kapsam genişlemesini önler ve okuyucu beklentilerini yönetir.

```
IN SCOPE / KAPSAM DAHİLİ:
1.
2.
3.

OUT OF SCOPE (explicitly) / KAPSAM DIŞI (açıkça):
1.
2.
3.
```

---

## 4. Executive Summary v0 / Yönetici Özeti v0

**EN:** Draft the executive summary NOW, before any research. This forces clarity about what the report is trying to say. It will be wrong — and that is fine. Update it as each phase completes. The final executive summary should differ substantially from v0.

**TR:** Yönetici özetini ŞİMDİ, herhangi bir araştırma yapmadan önce taslak olarak yazın. Bu, raporun ne söylemeye çalıştığını netleştirir. Yanlış olacak — bu sorun değil. Her aşama tamamlandıkça güncelleyin. Son yönetici özeti v0'dan önemli ölçüde farklı olmalıdır.

```markdown
## Executive Summary v0 — [Date]

**Context / Bağlam:** [1 sentence]

**Key question / Temel soru:** [1 sentence]

**Anticipated findings / Beklenen bulgular:** [2-3 bullets — your current hypothesis]
- •
- •
- •

**Anticipated recommendations / Beklenen öneriler:** [2-3 bullets]
- •
- •
- •

**Scope note / Kapsam notu:** This report does NOT address: [list]
```

---

## 5. Commissioner Sign-off / Komisyoncu Onayı

**EN:** Record explicit scope approval before proceeding. A verbal agreement does not count — get written confirmation.

**TR:** İlerlemeden önce açık kapsam onayını kaydedin. Sözlü anlaşma geçerli değildir — yazılı onay alın.

```
Scope approved by / Kapsamı onaylayan: ___
Date / Tarih: ___
Any scope modifications requested / İstenen kapsam değişiklikleri: ___
```
